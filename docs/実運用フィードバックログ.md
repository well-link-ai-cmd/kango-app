# kango-app 実運用フィードバックログ

実運用（試験運用）で気づいた事象を、その場で1行から記録するログ。
**改修判断はこのログの蓄積から行う**（`feedback_optimization_timing` 原則：工学最適化は実運用データが揃ってから）。

## 記入ルール

- 1事象 = 1エントリ。完璧に書かなくていい。**原文（実際の入出力の断片）を残すことが最優先**
- 個人情報は書かない（利用者名→「利用者A」等にマスキングしてから記録）
- 週1回（または訪問記録をまとめて書いた日）に見返して、パターン化した事象に ★ を付ける
- ★が付いた事象が改修候補。改修時は `tests/prompts/soap/cases.json` にテストケースを追加してから直す（テストハーネス運用）

## エントリのテンプレ

```
### YYYY-MM-DD 【機能名】事象の一言
- 分類: 誤変換 / 反映漏れ / 創作(ハルシネーション) / 文体ズレ / 長さ / UI / その他
- 何が起きたか:
- 入力の断片（マスキング済み）:
- 出力の断片:
- 期待していた挙動:
- 暫定対応: （手で直した内容）
```

## 改修サイクルとの接続

1. ★事象が3件以上たまる → 改修セッションを起動（「kango-appの続き」でproject-resume）
2. 改修前に `tests/prompts/soap/` でベースライン取得（baseline-YYYY-MM-DD.json）
3. プロンプト改修 → 全ケース再実行 → diff を目視レビュー
4. 対応済みエントリには「→ 対応済 YYYY-MM-DD（コミット/PR）」を追記。ログからは消さない
