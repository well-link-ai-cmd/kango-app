# 機能別 詳細レビュー（2026-06-01）

対象: kango-app（AI訪問看護記録アシスト）／レビュー観点: 役割・良い点・リスク・改善提案
凡例: 🟢 良好 ／ 🟡 改善余地 ／ 🔴 要対応（データ/安全に関わる）

---

## 0. 総評

**アーキテクチャは健全**。Next.js(App Router) + Supabase(Postgres+Auth+Storage) + Claude。
- データ書き込みはクライアントから `lib/storage.ts` 経由で anon key を使い、**RLSをセキュリティ境界**にしている（service-role keyをクライアントに置いていないのは◎）。
- AI呼び出しはサーバー側APIルート（11本）に集約し、`lib/ai-client.ts` で**エラー分類・自動リトライ・タイムアウト・Prompt Caching・vision**を一元化。
- AI責任分界（点数判定・宛先選定などは人間入力、ドラフトのみAI）が**全機能で明文化**されており、医療プロダクトとして統制が効いている。

**横断的に効く弱点（詳細は §9）**
1. 🔴 **保存失敗が画面に出ない**：`storage.ts` の保存系の多くが `console.error` のみで握りつぶす（`saveRecord` 以外）。失敗してもユーザーは「保存できた」と思い込むリスク。
2. 🟡 **テストが SOAP プロンプトのみ**。報告書・計画書系に回帰テストがない。
3. 🟡 **マルチテナント化に伴う後始末**：allowed_users 管理UIが空洞化、Storage RLSのorg分離が未完（本日のStage 3bで対応予定）。
4. 🟡 Next.js 16 の `middleware`→`proxy` deprecation 未対応。

## 1. SOAP生成（`app/api/soap`）— 🟢 中核機能・完成度高

**役割**: 口頭メモ＋S入力＋患者コンテキスト → S/O/A/P を Haiku 4.5 で生成。

**良い点**
- 🟢 tool_use でJSON強制＋`temperature 0.2` で出力安定。
- 🟢 S情報ポリシーが厳密（sInputがある時のみSにpassthrough、誤変換補正のみ。口頭メモ/過去記録からのS抽出禁止だがA/Pの判断材料には使う）。
- 🟢 音声誤変換の補正リストを継続メンテ（閉眼/肉芽/胃瘻/咳嗽/洗髪…）。
- 🟢 Few-shot 3例＋話者分類、Prompt Caching（1h・約14,000トークン）で**入力コスト約66%減**。
- 🟢 テストハーネス（cases.json／run.ts／quality-gate）整備済。

**リスク/改善**
- 🟡 過去SOAP参考化（1e7527c）の後方互換変換（`normalizeInitialSoap`）は実データでの取りこぼし確認を継続。
- 🟡 questions API は廃止済（alertsに集約）。設計判断としては妥当だが、CLAUDE.md にまだ「questions」の記述が残るので整理を。
- 🟡 PT用SOAP（30分・分岐）は未実装のまま要望に残っている。

## 2. 看護内容リスト（`app/api/nursing-contents/*`）— 🟢 UX改善済

**役割**: SOAP群を分析し、看護ケア項目の**追加候補・削除候補**を対話的に提示（extract/diff/appointments）。

**良い点**
- 🟢 PR #12 で「AIで整え直す(refine)」を撤去し、候補ごとに 採用/編集して採用/却下 の3アクションに統一。実運用に耐える粒度。
- 🟢 採用・却下がリアルタイムに画面反映。

**リスク/改善**
- 🟡 抽出系はAPIが候補を返し、**保存はクライアント**（`saveNursingContents`）。その保存が失敗しても握りつぶす（§9-1）。候補採用→保存失敗時に候補が消えて見える事故の可能性。
- 🟡 appointments（予定抽出）の誤抽出時のフォールバックUXは実運用FB待ち。

## 3. 褥瘡計画書（`app/api/pressure-ulcer-plan`）— 🟢 責任分界が明確

**役割**: 自立度・OHスケール・危険因子・DESIGN-R入力 → 看護計画5軸をHaikuで下書き。写真添付対応（010）。

**良い点**
- 🟢 DESIGN-R/危険因子/自立度は**人間入力（AI禁止）**、5軸テキストのみAI下書き、と分界が明快。
- 🟢 共通コンポーネント `PressureUlcerPlanForm`（new/edit/copy）で再利用性高い。
- 🟢 下書き/確定の2段階保存、OHスケール自動計算、判定ヘルプ完備。
- 🟢 写真はvisionで読む（4675e6e系）。本日の画像圧縮でStorageコストも最適化。

**リスク/改善**
- 🟡 `savePressureUlcerPlan` も保存失敗を握りつぶす（§9-1）。確定保存は特に失敗をユーザーに伝えるべき。
- 🟡 写真のEXIF（撮影日時・GPS）除去は未確認。個人情報・所在地が写真メタに残る懸念 → 圧縮時にcanvas経由で再エンコードするため**実はEXIFは落ちる**（本日の圧縮の副次効果）。ただし600KB以下や圧縮スキップ時は残るので、明示的な方針化を推奨。

## 4. 看護計画書 / NANDA（`app/api/nursing-care-plan/*`）— 🟢 設計の作り込み厚い

**役割**: 議事録＋直近SOAP＋active_plan → 課題ラベル候補(Sonnet) → OP/TP/EP一括生成(Sonnet) → 評価ドラフト(Haiku)。

**良い点**
- 🟢 創造系=Sonnet / 要約系=Haiku のモデル使い分けが合理的（評価をHaiku降格でコスト80%減）。
- 🟢 `NursingCarePlanIssue` を Discriminated Union（nanda/freeform）でモデル化、後方互換・コピペ取込み対応。
- 🟢 統合textareaUI（ラベル＋大textarea）でカイポケへのコピペ運用に最適化。

**リスク/改善**
- 🟡 Sonnetルートは `maxDuration=300` 設定済だが、`ai-client` 既定タイムアウトは30s。長文生成時は `timeoutMs` を明示的に上げているか要確認（現状5〜15sなら問題なし）。
- 🟡 issues の JSONB は型が緩い（`issueFromRow` で広く受ける）。壊れたデータ混入時の表示崩れ防止に最低限のバリデーションがあると安心。

**リスク/改善**

## 8. 認証・アクセス制御 / マルチテナント（`components/AuthGate.tsx`・`migration 011`）— 🟢 本日刷新

**役割**: Googleログイン → 所属事業所(membership)でアクセス判定 → 未所属はオンボーディング。

**良い点**
- 🟢 本日 org_id + RLS で**事業所ごとの完全分離**を導入。1デプロイで複数事業所。
- 🟢 セルフ登録（事業所作成／参加コード参加）、Googleログインのみで追加メール登録不要。
- 🟢 011未適用時はレガシー方式にフォールバックし、移行を安全に。

**リスク/改善（本日の引き継ぎ）**
- 🔴 **本番適用前にRLS分離の検証必須**（別アカウントで他事業所の患者が0件であること）。Supabase Branchingでの事前検証を強く推奨。
- 🟡 旧 `allowed_users` 管理UI（app/admin）は**アクセス制御の実権を失った**（membershipが正）。誤解防止のため membership 管理UIへ置換予定（emailを出すにはサーバールートが必要）。
- 🟡 `app_settings.org_password`・setup/check-access は当面レガシーとして残置。整理タイミングを別途。

## 9. AI基盤・横断（`lib/ai-client.ts` ほか）

**良い点**
- 🟢 `AiError`/`classifyAiError` で6分類、overloaded/rate_limitのみ指数バックオフ自動リトライ（最大2回）、timeoutは即fail。**画面に出す日本語文言が「次の一手」付き**で秀逸（PR #14）。
- 🟢 Prompt Caching（1h TTL）、vision、tool_use、haiku/sonnet切替を一箇所に集約。
- 🟢 失敗リクエストは非課金のためリトライのコスト増ゼロ。

**横断的な改善提案（優先度順）**
1. 🔴 **保存失敗のUI通知**：`savePatient`/`saveNursingContents`/`savePressureUlcerPlan`/`saveNursingCarePlan`/`saveVisitReport`/`saveInfoProvision`/`addPatientTodo` を、失敗時に例外を投げる（または成功可否を返す）よう統一し、各ページで「保存に失敗しました」を表示。**マルチテナント化でRLS拒否の経路が増える分、握りつぶしの危険度が上がっている**。
2. 🟡 **回帰テスト拡充**：報告書・計画書系にもSOAP同様の軽量スナップショットテスト。
3. 🟡 **Storageのorg分離（Stage 3b）**：写真パスを `org_id/...` に、Storage RLSをorgプレフィックス判定へ、既存ファイルの一度きり移行スクリプト。
4. 🟡 **管理UIのmembership化**＋ allowed_users/org_password のレガシー撤去。
5. 🟡 **Next.js 16 対応**：`middleware.ts`→`proxy.ts`。
6. 🟢 コスト最適化は良好（月数千円試算、SOAPキャッシュで入力66%減、本日の画像圧縮でStorage 1/5〜1/10）。利用拡大時は他高頻度ルートへのキャッシュ展開を検討。
