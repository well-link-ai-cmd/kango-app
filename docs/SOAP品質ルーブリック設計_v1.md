## 1. 目的

1. **回帰検知**: プロンプト改修で品質が上がったか下がったかを、感覚でなくスコアで判定する
2. **営業材料**: 「記録の質」を定量的に語れるようにする（展開時の差別化根拠）
3. **改修の優先順位付け**: 低スコア観点＝次に直すべき箇所、が自動で見える

LLM-judge は看護師レビューの**代替ではない**。回帰検知用の一次フィルタであり、最終品質判断は従来どおり看護師が行う。

## 2. ルーブリック6観点

| # | 観点 | 判定基準 | 検査対象 |
|---|------|---------|---------|
| R1 | **O純度** | Oに主観・評価語（「良好」「安定している」等の判断語、推測）が混入していない。観察事実・数値・実施内容のみ | O欄 |
| R2 | **S真正性** | Sは利用者・家族の言葉のみ。話者ラベル（本人/妻/娘等）が保持され、簡略化・統合・創作がない | S欄 vs sInput |
| R3 | **A根拠性** | Aの評価・解釈がS/Oに書かれた事実から導かれている。入力にない事実の創作・根拠の飛躍がない | A欄 vs S/O/入力 |
| R4 | **P接続性** | PがAの課題・S情報の訴えに対応し、次回訪問で観察・実行可能な具体形になっている（「継続」一語で終わらない） | P欄 vs A/S |
| R5 | **監査耐性** | 実地指導に耐える表現。断定的診断（医師権限）を書かない。ケアプラン・看護計画と矛盾しない。点数判定・区分選定をしていない | 全欄 |
| R6 | **継続性** | 前回記録がある場合、前回Pの計画事項が今回O（観察結果）またはA（評価）で回収されている | 全欄 vs previousRecords |

### 採点方式

- 各観点 **0/1/2 の3段階**（0=明確な違反あり / 1=軽微な問題 / 2=合格）
- **根拠引用を必須**（judgeは違反箇所の原文引用なしに0/1を付けられないスキーマにする）
- R6は前回記録がないケースでは `null`（採点対象外）
- ケーススコア = 適用観点の合計 / 満点。スナップショット全体で観点別平均を算出

## 3. Judge実装方針

| 項目 | 決定 | 理由 |
|------|------|------|
| Judgeモデル | **Claude Sonnet**（生成はHaiku 4.5） | ai-record-tools-design「例生成は実行モデルの一つ上」と同じ原則。判定はHaikuより一段上で |
| temperature | 0.0 | 判定の再現性優先 |
| 出力形式 | Tool use強制（JSON schema: 観点ごとに `score` + `evidence` + `reason`） | ブレ・パース失敗防止 |
| 観点の与え方 | 6観点を1コールで採点（観点ごとの独立プロンプトはコスト増の割にブレ低減効果が薄い。ブレが問題化したら分割） | コスト |
| Few-shot例の扱い | judgeには生成用Few-shotを**見せない** | 「Few-shotに似ているか」で採点する過学習判定を防ぐ |
| Judgeへの入力 | 入力素材（rawInput/sInput/previousRecords/carePlan）＋生成SOAP。生成プロンプト本文は渡さない | 「指示に従ったか」でなく「記録として良いか」を判定させる |

### コスト試算

8ケース × 1コール（入力 ~3k + 出力 ~1k トークン、Sonnet $3/$15）≈ **$0.10/回**。改修サイクルごとに実質無料で回せる。

## 4. ファイル構成（実装時）

```
tests/prompts/soap/
├── rubric.ts        # 6観点の定義・判定基準・スキーマ（単一ソース）
├── judge.ts         # snapshot JSON → ケースごとにSonnet judge → judge-{日付}.json + サマリ表示
└── CHANGELOG.md     # 改修ごとにルール合格数 + judgeスコアを併記（既存運用に列追加）
```

- 実行フロー: `run.ts` でsnapshot生成 → `quality-gate.ts`（ルールベース・誤変換等のhard-fail）→ 合格分を `judge.ts` に投入
- **2層の役割分担**: 機械判定できるもの（誤変換残存・S流入・passthrough失敗）は既存quality-gate.tsのまま。judgeは「質」の観点（R1〜R6）だけを見る
- baseline運用: 改修前スナップショットのjudgeスコアを固定 → 改修後と観点別diff → CHANGELOG 1行記録

## 5. 生成プロンプトへの反映（judge導入後）

1. 初回judge実行で観点別スコアを取得（現状把握）
2. 低スコア観点に対応するルールを `app/api/soap/route.ts` の systemPrompt に追記
   - 強調語（「特に重要」等）は既存3項目上限ルールを崩さない（ai-record-tools-design 落とし穴9）
   - 改修ごとに promptHash・スコア差分をCHANGELOGへ
3. R6（継続性）が低い場合は、前回P→今回O/A回収を促す指示 or Few-shot例1件追加を検討

## 6. 既知のリスクと対処

| リスク | 対処 |
|-------|------|
| judge自身の判定ブレ | temperature 0＋根拠引用必須＋3段階の粗い尺度（5段階にしない） |
| judgeの甘め判定（LLMは合格を出しやすい） | スキーマに「violations（違反列挙）を先に書かせてからscore」の順序を強制（自己チェック・スキャフォルド構造の応用） |
| R5（監査耐性）の基準曖昧さ | v1では「断定的診断」「点数判定・区分選定の混入」の2点に絞る。実地指導観点の拡張は看護師FBを得てから |
| スコアが営業材料として独り歩き | 外部提示時は「自社ルーブリックによる内部評価」と明記。第三者評価ではない |

## 7. 実装工数見込み

- rubric.ts + judge.ts: 1セッション（既存run.ts/quality-gate.tsのパターン踏襲）
- 初回ベースライン取得＋CHANGELOG整備: 同セッション内
- プロンプト反映（§5）: スコア次第で別セッション
