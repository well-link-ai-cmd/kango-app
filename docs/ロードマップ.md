## Phase 2: マルチテナント化（本運用前に必ず実施）

**重要**：シングルテナントのまま本運用を開始すると、後からマルチテナント化が大仕事になる。本運用前に必ずやる。

### アーキテクチャ選択
**B. 単一Supabaseプロジェクトでマルチテナント（論理分離）を採用**

- 理由：
  - Supabaseコスト固定（月$25）→ 事業所数が増えても1本でOK
  - 機能アップデートが1回で全事業所反映
  - SaaS化前提なら必須

### 実装内容
```sql
-- 新規テーブル
organizations (id, name, plan, created_at, ...)
organization_members (org_id, user_id, role)  -- 管理者/スタッフ

-- 既存テーブルに org_id を追加
patients (..., organization_id)
soap_records (..., organization_id)
nursing_contents (..., organization_id)
-- 今後追加する pressure_ulcer_plans, visit_reports, info_provisions すべて org_id を持つ
```

### RLS設計
- ログインユーザーの所属 org_id と一致する行のみ見える
- 他事業所のデータが絶対に漏れない設計
- **テスト必須**：RLS設計ミスは他事業所データ漏洩の致命傷

### 検討外の選択肢
- **A. 事業所ごとにSupabaseプロジェクト別々**：5事業所以下ならアリだが、運用コストが事業所数に比例するので却下
- **C. コードベース複製**：運用地獄なので却下

## Phase 3: 自ステーション本運用（Supabase Pro移行）

### 移行項目

| 対策 | 内容 |
|---|---|
| **Pro移行（月$25）** | プロジェクト一時停止なし、日次バックアップ、7日監査ログ |
| **IP制限** | 事業所WiFi・訪問端末以外からDBアクセス不可 |
| **監査ログ活用** | Supabase標準の7日分ログ＋アプリ側で独自操作履歴テーブル（永続保存） |
| **患者氏名の仮名化** | UUID管理、表示時のみ氏名復元 |
| **MFA強制** | Google OAuthに多要素認証必須化 |
| **退職者フロー** | アカウント即時停止の運用ルール |

### 厚労省ガイドライン準拠のチェックリスト
- [ ] 東京リージョン保存（国内保存原則）
- [ ] 通信暗号化TLS 1.3（Supabase標準）
- [ ] 保存時暗号化AES-256（Supabase標準）
- [ ] アクセス制御（RLS）
- [ ] 監査ログ（Pro plan）
- [ ] 定期バックアップ（Pro plan日次）
- [ ] インシデント対応手順書

## Phase 4: SaaS化・他事業所展開

### 収益モデル（叩き台）

| プラン | 月額 | 対象 |
|---|---|---|
| 小規模 | 5,000〜10,000円 | 患者50名・スタッフ5名まで |
| 標準 | 20,000〜30,000円 | 患者200名・スタッフ20名まで |
| 大規模 | 要相談 | 無制限 |

### 費用構造
- Supabase Pro：月3,800円（全事業所共通）
- Vercel：月$20 Pro plan（必要になったら）
- Claude API：従量課金（患者数×月間SOAP件数で試算）
- Stripe決済手数料：3.6%
- → **事業所1契約でSupabaseコストはペイ**、2事業所目以降はほぼ利益

### 実装項目
- 事業所登録フロー（サインアップ画面）
- 管理者招待・メンバー追加機能
- 事業所ごとの設定（看護内容テンプレ・プロンプト調整等）
- Stripe連携（月額サブスク課金）
- プラン制限（患者数・ユーザー数の上限チェック）
- 請求書発行

### IT導入補助金ベンダー連携
- 既存プロジェクト：kango-app + 診療報酬改定レーダー + ほうかんナビ のパッケージ化
- ベンダー経由で補助金活用しての導入提案
