# 看護計画書 AI自動生成ツール 手順書

> 責任者レビュー用ドキュメント。カイポケ（介護ソフト）の「訪問看護計画書」フォーマットに準拠して、AI下書き生成 → 看護師確認・修正 → カイポケへ項目別コピペまでの運用設計をまとめたもの。この手順をもとにAIプロンプト・DB・UIを設計する。

---

## 0. 前提：使用する様式

- **カイポケの「訪問看護計画書」フォーマットに準拠**
- 計画書タイプ：**介護保険 / 医療保険** の2種（書く中身は同じ、制度上の区分のみ）
- タイトル：**共通 / 看護 / リハ** の3種（UI上で選択）
- 「看護・療養の情報」「療養上の課題・支援内容」「衛生材料の情報」「備考」の4ブロック構成
- **本ツールはカイポケ転記前提**。カイポケの画面構成に完全準拠して出力する

---

## 0-1. 作成タイミング・更新サイクル

- **新規作成**：利用者の訪問開始時、または状態変化時
- **定期更新**：介護保険ケースは **6ヶ月ごと**（標準的な更新サイクル）、医療保険は必要に応じて
- **計画書の履歴保持**：新しい計画書は既存のものを複製（コピー）して修正する運用。過去計画書は削除せず履歴として残す
- **「有効版」の定義**：`plan_date` が最新 かつ `is_draft = false` のものを「現在有効な看護計画書」とする。SOAP生成時はこれを参照

## 1. 作成フロー（全体像）

```
患者情報（基本情報・主治医・ケアマネ）
  ↓
ケア内容リスト（登録済み実施ケア項目）
  ↓
直近SOAP記録（参照コンテキスト）
  ↓
Step 1: 計画書タイプ・タイトル選択（看護師手入力）
  ↓
Step 2: 看護・リハビリの目標（AI下書き → 看護師修正）
  ↓
Step 3: 療養上の課題・支援内容（AI下書き・行ごと → 看護師修正）
  ↓
Step 4: 衛生材料の情報（看護師手入力、AI補助なし）
  ↓
Step 5: 備考（AI補助可、任意）
  ↓
Step 6: 下書き保存 / 確定保存（2段階）
  ↓
Step 7: カイポケに項目別コピペ
  ↓
（数ヶ月経過後）
  ↓
Step 8: 評価（AI下書き・期間SOAPから生成 → 看護師確認）
  ↓
Step 9: 次回計画書を複製して修正
```

## 2. カイポケの項目（全項目）

### 2-1. 計画書タイプ（看護師手入力）
- テンプレート：介護 / 医療
- タイトル：共通 / 看護 / リハ

### 2-2. 利用者情報（既存データから自動流用）
- 利用者氏名・生年月日・住所
- 要介護認定の状況
- 居宅介護支援事業所・ケアマネ名（既存 `careManagers` 配列から）

### 2-3. 指示書と主治医（既存データから自動流用）
- 医療機関名称
- 主治医氏名・診療科
- 住所・電話番号（既存 `doctors` 配列から）

### 2-4. 看護・療養の情報（署名印字項目）
- 職員氏名・肩書き（看護師入力、履歴保持）
- 作成年月日

### 2-5. 看護・リハビリの目標（AI下書き可）
- 自由記述、**3000字**
- 患者情報・ケア内容・直近SOAPから生成
- 看護師が確認・修正して確定

### 2-6. 療養上の課題・支援内容（複数行）
各行：
- No（自動採番）
- 年月日（作成日プリセット）
- **課題・支援内容**：2500字、**AI下書き可**
- **評価**：2500字、**AI下書き可**（期間SOAP総合評価、看護師確認必須）

### 2-7. 衛生材料の情報（看護師手入力、AI禁止）
- 衛生材料が必要な処置の有無（有り/なし）
- 処置の内容：3000字
- 衛生材料（種類・サイズ）等：3000字
- 必要量：3000字

### 2-8. 備考（AI補助可・任意）
- 自由記述、3000字

### 2-9. 作成者
- 作成者①氏名・職種（必須）
- 作成者②氏名・職種（任意）

## 3. AI責任分界（明文化）

### ✅ AI下書き可能領域

| 項目 | 根拠 |
|---|---|
| 看護・リハビリの目標 | 患者情報・ケア内容・SOAPから妥当な目標文言を下書き可 |
| 療養上の課題・支援内容（課題列） | 直近SOAPからの課題抽出・支援内容文言化 |
| **評価列**（期間SOAP総合評価） | **一定期間のSOAP記録からの経過整理は AI が得意。ただし「改善/悪化」の医学的判定は看護師最終確認** |
| 備考 | 要約・文体調整 |

### ❌ AI禁止領域（看護師手入力）

| 項目 | 理由 |
|---|---|
| 計画書タイプ（介護/医療） | 制度判断・医療保険/介護保険の適用区分 |
| タイトル（共通/看護/リハ） | 算定区分の選択、事業所運用判断 |
| 衛生材料の種類・サイズ・必要量 | 医療材料選定は医師指示・看護師判断、誤選定は医療安全上重大 |
| 作成者・署名情報 | 記名押印は看護師本人の責任 |
| 主治医・ケアマネ情報 | 外部データ、手動確認が必要 |

### AI出力マーカー
すべてのAI下書き出力に **「※AI下書き。看護師確認必須」** を付与。
メタ情報として `ai_model` `ai_prompt_version` `ai_generated_at` を保存。

## 4. 入力コンテキストの役割分離

AI生成API に渡す入力フィールドと、その役割：

| 入力 | 役割 | 用途 |
|---|---|---|
| 患者情報（年齢・診断・要介護度） | Judgment-only | 目標・課題の妥当性判断材料 |
| ケア内容リスト | Extraction | 実施している支援内容の抽出源 |
| 直近SOAP 5件（generate時）/ 期間全SOAP（evaluate時） | Extraction | 課題・経過の抽出源 |
| 前回の看護計画書（複製元がある場合） | Passthrough + Judgment | 前回の課題を引き継ぎ、評価で変化を判定 |
| 主治医指示・ケアマネ計画（過渡期の旧 `carePlan` 欄） | Judgment-only（過渡期のみ） | 外部情報、移行完了後は非参照 |

## 5. 評価機能の詳細仕様

### 5-1. 評価の対象
- 「療養上の課題・支援内容」の**各行（課題ごと）**に対する評価
- 課題ごとに独立したAPIコール、または一括評価モードで複数同時生成

### 5-2. 評価の期間
- **デフォルト**：「前回計画書 `plan_date` 〜 今日」自動設定（前回がなければ直近3ヶ月）
- **オーバーライド**：1ヶ月 / 3ヶ月 / 6ヶ月 / 12ヶ月 / カスタム日付範囲
- 介護保険の標準更新サイクルに合わせやすいUI

### 5-3. 評価の出力構造

1. **経過サマリ**（期間内のSOAPから該当課題に関する記録を整理）
2. **変化のポイント**（期間開始時と現在の状態変化、ADL変化、症状変化）
3. **所見下書き**（継続/改善/達成/見直し の候補文言、**最終判定は看護師**）

### 5-4. 利用条件（UI制御）

- 該当期間のSOAP記録が **3件未満** の場合、評価ボタンを無効化し、ガイド表示：
  - 「記録が3件以上貯まると評価できます（現在 N件）」

### 5-5. 一括評価モード

- 課題が複数行ある場合、デフォルトは「全課題を一括評価」（Tool use で配列出力、1回のAPIコール）
- オプションで「個別に評価」も選択可（特定課題のみ再生成したい時）

## 6. DB設計

### テーブル：`nursing_care_plans`

```sql
CREATE TABLE nursing_care_plans (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  patient_id TEXT NOT NULL REFERENCES patients(id) ON DELETE CASCADE,

  -- 監査情報
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  created_by UUID REFERENCES auth.users(id),
  user_id UUID REFERENCES auth.users(id),

  -- 基本情報
  plan_date DATE NOT NULL,
  plan_type TEXT CHECK (plan_type IN ('介護','医療')) DEFAULT '介護',
  plan_title TEXT CHECK (plan_title IN ('共通','看護','リハ')) DEFAULT '共通',
  is_draft BOOLEAN DEFAULT TRUE,

  -- 作成者
  author_name TEXT,
  author_title TEXT,
  author2_name TEXT,
  author2_title TEXT,

  -- 看護・リハビリの目標（3000字、AI下書き可）
  nursing_goal TEXT,

  -- 療養上の課題・支援内容（複数行、JSON配列）
  -- [{ no: 1, date: "2026-04-22", issue: "...", evaluation: "...", evaluated_at: "..." }, ...]
  issues JSONB DEFAULT '[]'::jsonb,

  -- 衛生材料の情報
  has_supplies BOOLEAN DEFAULT FALSE,
  supply_procedure TEXT,
  supply_materials TEXT,
  supply_quantity TEXT,

  -- 備考
  remarks TEXT,

  -- AI生成メタ情報（監査用）
  ai_model TEXT,
  ai_prompt_version TEXT,
  ai_generated_at TIMESTAMPTZ
);

CREATE INDEX idx_nursing_care_plans_patient_id ON nursing_care_plans(patient_id);
CREATE INDEX idx_nursing_care_plans_patient_date ON nursing_care_plans(patient_id, plan_date DESC);

-- RLS（褥瘡計画書と同じパターン）
ALTER TABLE nursing_care_plans ENABLE ROW LEVEL SECURITY;
-- SELECT/INSERT/UPDATE/DELETE ポリシー（認証済みユーザー）
```

## 7. 既存 carePlan 欄の扱い（過渡期）

### 7-1. 現状
- `patient.carePlan` フィールドに看護計画を自由記述で貼っている運用
- この欄は看護計画書feature完成後は新規入力不要となる

### 7-2. 移行方針

**段階1：新feature リリース時**
- 新規患者作成フォーム：`carePlan` 欄を撤去
- 既存患者編集フォーム：`carePlan` 欄を**読み取り専用**で表示
- 看護計画書画面に「**旧ケアプラン欄の内容を移行**」ボタンを設置
  - ボタン押下で旧 carePlan のテキストが看護計画書の「備考」欄に下書きコピー
  - 看護師がそれを見ながら「看護・リハビリの目標」「療養上の課題」に手動で振り分け
  - **自動振り分けはしない**（内容の解釈ミスを避けるため）

**段階2：全患者移行完了後**
- `carePlan` カラムを DB から drop（マイグレーション）

### 7-3. SOAP生成プロンプトの参照優先順位（過渡期）

```
1. 看護計画書（確定版 = is_draft=false の最新 plan_date） ← あればこれだけ参照
2. carePlan（旧） ← 看護計画書がなく、carePlan に何か入っていれば補助参照
3. ケア内容リスト ← 常に参照
4. 推論（患者情報のみ） ← 上記すべてなければ
```

## 8. UI設計の原則

### 8-1. AI可 / AI禁止の視覚明示
- AI下書き可の欄：🤖 **AI下書き可** バッジ
- 看護師手入力欄：👤 **看護師記入必須** バッジ
- 同一フォーム上で混在しても看護師が迷わない

### 8-2. AI生成の2モード
- **ゼロから再生成**：既存の内容を破棄して新規生成
- **現在の内容を改善**：既存の内容を入力として AI に渡し、差分プレビューで表示 → 承認反映
  - 看護師の修正を上書きしないため、デフォルトは「改善」モード

### 8-3. 下書き / 確定 の2段階保存
- 下書き（`is_draft=true`）：SOAP生成の参照対象**外**
- 確定（`is_draft=false`）：SOAP生成で参照される
- UIで明示的に「確定保存」ボタンを設ける

### 8-4. カイポケへのコピペ導線
- 褥瘡計画書と同じく、各項目ごとに「コピー」ボタン
- テキスト形式でクリップボードへ

## 10. リリース後の確認項目（実運用フィードバック）

- AI出力に「追記・修正」した箇所（目標・課題・評価それぞれ）
- 不自然な表現・音声誤変換
- 評価機能での「期間が長すぎて要点ぼやける」などの事象
- カイポケへのコピペで書式崩れが起きないか
- 「現在の内容を改善」モードで看護師修正が保持されているか
- 一括評価 vs 個別評価、どちらが実運用で使いやすいか

これらを集めた上で、Prompt Caching や 2段階生成などの工学最適化を検討する（実運用前の先行最適化は行わない）。
