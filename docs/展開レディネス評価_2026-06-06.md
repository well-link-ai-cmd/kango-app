## 0. 総合判定（エグゼクティブサマリ）

**結論：技術的な土台（マルチテナント論理分離）は実装済みで質も高い。ただし、他事業所へ「商用提供」するには法務・運用・一部のセキュリティに展開ブロッカーが残っており、現状のままの外部展開は不可。**

- ✅ **強い**：DB のテナント分離（org_id + RLS）、認証（Google OAuth + 招待コード/メール招待）、AIエラー耐性、管理画面（メンバー/権限/招待/評価周期）。
- 🔴 **致命的（展開ブロッカー）**：①Storage（画像）のRLSがテナント分離されていない ②監査ログ（操作履歴）が存在しない ③利用規約/プライバシーポリシー/同意フローが存在しない ④医療情報の越境送信（Anthropic）に対する契約・同意の整理が未了。
- 🟠 **重要（商用運用に必須）**：バックアップ/復旧（Free planはPITR/日次バックアップなし）、問い合わせ窓口、課金、監視/エラートラッキング、メール招待の実配信。
- 🟡 **改善推奨**：CSP/Permissions-Policy、アイドルタイムアウト、レガシー認証（パスワード4文字・allowed_users）の撤去。

レディネス・スコア（10項目平均）：**約 5.0 / 10（＝「自ステーション本運用」相当。外部商用展開は未達）**。
ロードマップ（`docs/ロードマップ.md`）の Phase で言えば **Phase 2 完了・Phase 3 着手前**。本書の「クリティカルパス」が Phase 3〜4 の具体化に相当する。

## 2. 項目別 詳細所見（実コード根拠）

### 1) データ保存先・認証 — ◎ 良好
- **クラウドDB＝Supabase（PostgreSQL）**。ブラウザ/サーバ両用クライアントを `@supabase/ssr` で実装（`lib/supabase.ts`, `lib/supabase-server.ts`）。
- **認証＝Google OAuth**（`AuthGate.tsx:151` `signInWithOAuth({provider:"google"})`）。
- **テナント参加＝招待コード/メール招待**。組織作成 `create_organization()`、参加 `join_organization()`（`migrations/011:84-118`）、メール招待 `invite_member`/`accept_invites`（`migrations/013`）。
- 書き込み時の org_id 注入は `getCurrentUserAndOrg()`（`lib/storage.ts:18-33`）で一元化。読み取りは RLS 任せ。
- **所見**：設計は妥当。要件定義の「localStorage/シングルユーザー」記述は実装と乖離（ユーザー指摘どおり旧情報）。

### 2) テナント分離 — △ DBは堅牢だが Storage に分離漏れ（🔴ブロッカー）
- **DB（良）**：8データテーブル全てに `org_id`、RLSを `org_id in (select current_org_ids())` の4ポリシー（select/insert/update/delete）へ全面書換（`migrations/011:210-239`）。`current_org_ids()`/`is_org_admin()` を SECURITY DEFINER 化して RLS 再帰を回避。組織作成/参加は SECURITY DEFINER RPC のみ。**設計として正しい**。検証SQLも整備（`supabase/manual/011_verify.sql`）。
- **🔴 Storage（穴）**：画像バケット `patient-files` の RLS が **`bucket_id='patient-files' かつ authenticated` だけ**（`migrations/010:16-35`）。**org_id でスコープされていない**。つまり、別事業所のユーザーでもログインさえしていれば、バケット内の任意ファイルを read/write/delete できる。
  - 保存パスはランダムID（`generateId()`）なので「パスを知らなければ事実上アクセスされにくい」=セキュリティ・バイ・オブスキュリティに留まる。CLAUDE.md は「DB行は分離済みで実害は出ない設計」とするが、**Storage層自体はテナント境界になっていない**。
  - 対象画像にはケアプラン写真・**褥瘡創部写真**等が含まれ、医療情報。商用展開では真のテナント境界が必須。
- **🟡 レガシー表**：`allowed_users`/`app_settings` は authenticated 全読み取り（`migrations/002`）。011 で対象外（事業所横断の旧認証設定）。組織横断でメール/権限/旧orgパスワードが読めてしまうため、撤去または閉塞が望ましい。

### 3) バックアップ・復旧 — ✗ 仕組み未整備
- アプリ/リポジトリ内にバックアップ・エクスポート・復旧の仕組みは無い。
- Supabase **Free plan は日次自動バックアップ・PITR（ポイントインタイムリカバリ）が無い**。ロードマップも Pro 移行を Phase 3 に先送り（`docs/ロードマップ.md:86`）。
- **商用展開では事業所のデータ消失＝重大事故**。Pro 以上（日次バックアップ／PITR）＋ 復旧手順書（RPO/RTO 定義）が必須。

### 4) 医療コンプラ（3省2ガイドライン） — ✗ 大半未対応
「3省2ガイドライン」＝厚労省「医療情報システムの安全管理に関するガイドライン 第6.0版」＋ 経産省・総務省「医療情報を取り扱う情報システム・サービスの提供事業者における安全管理ガイドライン」。訪問看護ステーションも医療情報取扱事業者として対象。

| 要求領域 | 現状 | 評価 |
|---|---|---|
| アクセス制御 | RLS でテナント/権限分離（Storage除く） | ○（Storage要対応） |
| **監査ログ（誰がいつ何を閲覧/変更）** | **存在しない**。`user_id` は作成者列のみで操作履歴ではない | ✗ 致命 |
| 通信暗号化 | Vercel/Supabase の TLS（プラットフォーム標準）。HSTS は設定済(`next.config.ts`) | ○ |
| 保存時暗号化 | Supabase（AWS）標準のディスク暗号化に依存。アプリ/列レベル暗号化・仮名化は無 | △ |
| 認証強度 | Google OAuth。**MFA強制なし**。レガシーorgパスワードは**最小4文字**(`AuthGate`/`api/auth/setup`) | △ |
| **委託先管理（越境送信）** | SOAP・主病名等の医療情報を Anthropic API へ送信。非学習/保持/ZDR・同意の整理が**未了**（CLAUDE.md 明記） | ✗ |
| **BCP（非常時計画）** | 計画・手順書なし。バックアップも未整備 | ✗ |
| リスクアセスメント/規程 | 文書なし | ✗ |

- セキュリティヘッダは部分実装（`next.config.ts`：HSTS/X-Frame/X-Content-Type/Referrer。**CSP・Permissions-Policy は未設定**）。
- アイドルタイムアウト無（`sessionStorage` の `access_verified` でセッション内再認証スキップ。共用端末リスク）。
- アプリ層のレート制限/ブルートフォース対策なし（Supabase 側の標準のみ）。

### 5) 利用規約・プライバシーポリシー・利用契約 — ✗ 不在
- `app/`・`docs/`・`public/` に規約/ポリシー/同意の**いずれも無し**。同意チェックボックスもオンボーディングに無い（`AuthGate.tsx`）。
- 医療情報＋AI越境送信を伴うため、**利用者（患者・家族）への説明と同意**、**事業所との利用契約（SaaS規約・DPA/委託契約）**、**プライバシーポリシー**は展開の法的前提。

### 6) 問い合わせ機能 — ✗ 不在（要設計）
- 問い合わせ/サポート/フィードバックのルート・UI・メール送信（nodemailer/Resend等）は無い。現状は文言で「管理者にお問い合わせください」(`AuthGate.tsx:622`)。
- CLAUDE.md/バックログでも「問い合わせフォーム（他事業所展開時に必要・後日）」と明記。**本書で設計対象**（§4参照）。

### 7) 課金・請求 — ✗ 不在
- Stripe等の決済依存・コード・プラン/上限/従量メータリングは皆無。`organizations` に plan 列も無い。
- ロードマップ Phase 4 に料金案（5,000〜30,000円/月、患者数課金、Stripe手数料3.6%）はあるが未実装（`docs/ロードマップ.md:104-127`）。

### 補）監視・メール配信・オンボーディング
- **監視**：Sentry/DataDog 等なし。エラーは console / Vercel ログのみ。`lib/ai-error-response.ts` は UX 文言用で運用監視ではない。
- **メール招待**：`org_invites` に行を作るだけで**実メール送信は無い**。参加コード/リンクは管理者が口頭/他チャネルで伝える運用。
- **オンボーディング**：事業所作成→参加コード表示までは良好。**規約同意・初期ウィザードは無し**。

## 3. クリティカルパス（展開ブロッカー → 解除順）

外部事業所に「お金をもらって」提供する前に、最低限これらを解除する必要がある。上から順に依存関係が小さく着手しやすい。

```
[P0 法務・前提]
  L1. 利用規約 / プライバシーポリシー / 利用契約(DPA) 整備 ──┐
  L2. 医療情報の越境送信ポリシー確定（Anthropic 非学習・   │ ← 営業/契約のブロッカー
      保持・ZDR要否・患者/家族同意の運用） ────────────────┘

[P0 セキュリティ・データ保全]
  S1. Storage(patient-files) の org スコープ RLS 化 ＋ 既存パス移行
  S2. 監査ログ（操作履歴）テーブル＋記録＋閲覧
  S3. バックアップ/復旧（Supabase Pro：日次＋PITR）＋復旧手順書(RPO/RTO)

[P1 商用運用]
  O1. 問い合わせフォーム（事業所/送信者の自動付与・通知）
  O2. メール招待の実配信（Resend 等）
  O3. 監視・エラートラッキング（Sentry）＋ヘルスチェック
  C1. 課金・請求（Stripe＋プラン上限）  ← 収益化時

[P2 強化]
  H1. CSP/Permissions-Policy, アイドルタイムアウト, MFA強制
  H2. レガシー認証撤去（allowed_users / org_password / check-access/setup）
  H3. 患者氏名の仮名化・列レベル暗号化
```

## 4. 充足設計（各施策の方針）

### S1. Storage の org スコープ RLS 化（🔴最優先・低コスト高効果）
- **方針**：保存パスを `org_id` プレフィックス化（`<org_id>/<patientId>/<uuid>.jpg`）し、`storage.objects` の RLS を
  `(storage.foldername(name))[1]::uuid in (select current_org_ids())` でスコープ。
- **作業**：①`lib/storage.ts` のアップロードパス生成に org_id を付与 ②新 migration で 010 のポリシーを org スコープへ置換 ③既存ファイルの移行（パス付け替え＋DB JSONB の path 更新スクリプト、`supabase/manual/` に検証/ロールバックSQL同梱）。
- **リスク**：既存画像の移行が要注意。先に新規だけ org プレフィックスにし、移行は別ステップでも可。

### S2. 監査ログ（3省2ガイドライン要求）
- **方針**：`audit_logs(id, org_id, user_id, action, entity, entity_id, at, ip?, meta jsonb)` を新設。RLS は org スコープ＋**管理者のみ閲覧**、アプリからの **INSERT専用**（UPDATE/DELETE禁止＝改ざん防止）。
- **記録点**：患者の閲覧/作成/更新/削除、SOAP・各書類の生成/保存、AI送信（医療情報の越境）イベント、権限変更/招待。最小は「書込系＋AI送信」から。
- **実装案**：書込は `lib/storage.ts` の各 save に集約済みなので、ここに記録フックを差すのが低コスト。閲覧ログは段階導入。
- **保持**：ガイドライン目安に沿い長期保存（Supabase標準7日ログとは別の永続テーブル）。

### S3. バックアップ・復旧
- **方針**：Supabase **Pro 移行**（日次自動バックアップ＋PITR）。加えて、月次の論理エクスポート（pg_dump）をオフサイト保管する運用も検討。
- **文書**：復旧手順書（RPO/RTO、誰が・どの順で・どこから復元するか）、インシデント対応フロー。BCP の中核。

### L1/L2. 規約・ポリシー・越境送信（法務）
- **成果物**：`/terms`（利用規約）、`/privacy`（プライバシーポリシー）、事業所向け **利用契約（SaaS規約＋個人情報の取扱い/委託契約・DPA）**。`app/` にページ化し、**初回ログイン時に同意チェック**（`organizations` or `memberships` に `agreed_terms_at` 記録）。
- **越境送信**：Anthropic の「学習不使用・保持期間（最大30日）」規約の確認、必要に応じ **ZDR（Zero Data Retention）契約**、利用者・家族への説明と同意の運用設計。プライバシーポリシーに第三者提供（AI処理委託・国外移転）を明記。
- ⚠️ 法務文面は専門家レビュー前提。本プロジェクトでは**雛形作成＋アプリ組込み（同意フロー）**まで担当し、文面の最終確認は弁護士/社労士等へ。

### O1. 問い合わせフォーム（バックログの正式設計）
- **方針**：`app/contact` ＋ `app/api/contact`。送信時に **org_id・送信者（user email/name）・ロール・アプリ版数を自動付与**。本文＋カテゴリ（不具合/要望/その他）。
- **保存＋通知**：`inquiries` テーブルに保存（org スコープRLS）＋ 運営宛通知（メール（Resend）or Slack Webhook）。まずDB保存＋メール通知が堅実。
- **既存資産**：エラー耐性スキル（`.claude/skills/external-api-resilience`）に倣い、送信失敗時の文言と再送を踏襲。

### O2. メール招待の実配信
- **方針**：現状DB行のみ → **Resend 等で実送信**（招待リンク/参加コードをメール）。Supabase Edge Function or Next API どちらでも可。`invite_member` 後に送信をフック。

### O3. 監視
- **方針**：Sentry 導入（フロント＋APIルート）。`/api/health` ヘルスチェック。AI送信失敗・保存失敗の集約。

### C1. 課金（収益化時）
- **方針**：`organizations.plan` ＋ Stripe（サブスク）＋ 上限チェック（患者数/ユーザー数）。Webhook で状態同期。請求書は Stripe 発行を流用。**他ブロッカー解除後**で良い。

### H 系（強化）
- CSP/Permissions-Policy を `next.config.ts` に追加、アイドルタイムアウト（一定無操作で sessionStorage 破棄＋再認証）、Google MFA 強制、レガシー認証（`allowed_users`/`org_password`/`api/auth/*`/`api/admin/*`）の撤去、患者氏名の仮名化。

## 5. 推奨ロードマップ（優先順位）

| 順 | 施策 | 区分 | 規模感 | 効果 |
|---|---|---|---|---|
| 1 | **S1 Storage org RLS** | セキュリティ | 小〜中 | テナント境界の致命的穴を塞ぐ |
| 2 | **S2 監査ログ（書込＋AI送信）** | コンプラ | 中 | ガイドライン必須要件 |
| 3 | **L1/L2 規約・同意・越境ポリシー** | 法務 | 中（文面は外部レビュー） | 契約/営業の前提 |
| 4 | **S3 Pro移行＋復旧手順** | 運用 | 小（設定）＋文書 | データ保全・BCP |
| 5 | **O1 問い合わせフォーム** | 運用 | 小〜中 | サポート窓口 |
| 6 | O2 メール招待実配信 / O3 監視 | 運用 | 小 | 運用品質 |
| 7 | H1/H2 ヘッダ・タイムアウト・レガシー撤去 | 強化 | 小〜中 | 堅牢化 |
| 8 | C1 課金 | 収益 | 中〜大 | 収益化（最後でよい） |

> 技術だけで「明日から有償展開」に最短到達するなら **1→2→4→5**（＝Storage穴を塞ぎ、監査ログとバックアップと窓口を持つ）。並行して **3（法務）** を外部レビューと進めるのが現実的。

## 6. 次アクション提案

本書の優先順位で合意できれば、**まず S1（Storage の org スコープ RLS 化）から実装**するのが最も費用対効果が高い（致命的なテナント分離の穴・小さな差分・既存挙動を壊しにくい）。続けて S2 監査ログ、S3/L 系へ。

実装に入る際は、どのトラックから着手するかをご指示ください（推奨は S1）。
